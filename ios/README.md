# SilverCare iOS

This is the iOS migration workspace for 银龄智护.

## Layout

- `SilverCareiOS.xcodeproj`: generated Xcode project.
- `project.yml`: XcodeGen source of truth.
- `SilverCareiOS/`: SwiftUI app shell, WKWebView bridge, iOS services.
- `Native/SilverCareMNNRuntimeABI.h`: C ABI expected by the dynamic iOS MNN runtime bridge.
- `Native/SilverCareMNNTTSRuntimeABI.h`: C ABI expected by the dynamic iOS MNN TTS runtime bridge.
- `Sources/SilverCareCore/`: platform-neutral Swift core for navigation, inquiry routing, offline vision interpretation, offline model readiness/download management, ASR correction, fall confirmation, task guidance, and smart refresh.
- `Tests/SilverCareCoreTests/`: XCTest coverage for the Swift core.
- `Tools/verify_core.swift`: command-line verifier for machines that have Swift but not a full working SwiftPM/Xcode installation.

## Generate the project

```sh
cd ios
xcodegen generate
```

## Verify the core on this machine

Full Xcode and SwiftPM are available on the current machine. Run:

```sh
cd ios
swift test
```

Expected: all Swift Package tests pass.

## Simulator automation

From the repository root, run:

```sh
npm run test:ios:sim
```

This is the default iOS simulator gate. It regenerates the Xcode project, runs
JavaScript checks, JavaScript unit tests, Swift Package tests, simulator-safe
Xcode unit and UI tests, builds and installs the app, launches the simulator app,
verifies local benchmark reports, checks recent app logs, and captures
`ios/build/simulator-automation/silvercare-home.png`. The default Xcode test
selection includes `SilverCareiOSTests` and
`SilverCareiOSUITests/SilverCareiOSUITests`; use `SILVERCARE_RUN_FULL_XCODE_TESTS=1`
to run every Xcode test target, or `SILVERCARE_SKIP_XCODE_TESTS=1` for a quick
local smoke pass.

## Vosk ASR runtime binding

iOS local ASR now mirrors the Android path structurally:

1. Record microphone audio as 16kHz mono PCM.
2. Load `vosk-model-small-cn-0.22` from app support storage.
3. Dynamically load Vosk C API symbols such as `vosk_model_new` and `vosk_recognizer_new`.
4. Feed PCM chunks to Vosk and parse the final JSON `text`.

The official Vosk installation page states that the iOS build is available on request from Alpha Cephei: [Vosk Installation](https://alphacephei.com/vosk/install). The upstream iOS demo README also says the demo requires a Vosk-API library build: [vosk-api ios README](https://github.com/alphacep/vosk-api/blob/master/ios/README). This workspace now includes `ios/Native/Vendor/libvosk.xcframework`, generated by `ios/Tools/install_vosk_runtime_artifact.sh` from a static Vosk iOS xcframework into dynamic `libvosk.dylib` slices for iPhoneOS and arm64 simulator. For production release, replace or verify the artifact against the official supplier and license requirements.

Accepted runtime locations are:

```text
SilverCareiOS.app/Frameworks/vosk.framework/vosk
SilverCareiOS.app/Frameworks/libvosk.framework/libvosk
SilverCareiOS.app/Frameworks/libvosk.dylib
SilverCareiOS.app/libvosk.dylib
```

If the supplied artifact is a static `libvosk.a` or static `libvosk.xcframework`, run:

```sh
ios/Tools/install_vosk_runtime_artifact.sh /path/to/libvosk.xcframework
```

The script converts static archive slices into `ios/Native/Vendor/libvosk.xcframework`
with dynamic `libvosk.dylib` binaries by linking only the Vosk C API symbols
used by the app. Dynamic artifacts are copied directly. For simulator debugging,
`LocalVoskASRRuntime` also checks `SILVERCARE_VOSK_LIBRARY` for an absolute
dylib/framework binary path. If `libvosk.a` is linked directly into the app
executable instead, the runtime probes the main image symbols via `dlopen(nil)`.

## Native runtime verification

The migration includes an explicit runtime contract gate:

```sh
node tools/check-ios-native-runtime.mjs
```

Default mode verifies the Swift bridge, MNN ABI header, MNN TTS ABI header, Vosk
dynamic loader, and known artifact locations while reporting whether iOS
MNN/MNN TTS/Vosk binaries are currently bundled. The current workspace has the
SilverCare MNN wrapper, SilverCare MNN TTS wrapper, and dynamic Vosk xcframework
available. Final offline runtime packaging requires strict mode to pass:

```sh
SILVERCARE_REQUIRE_IOS_NATIVE_RUNTIME=1 node tools/check-ios-native-runtime.mjs
```

`SILVERCARE_REQUIRE_IOS_NATIVE_RUNTIME=1` currently passes for the source-tree
runtime artifacts. Real offline parity still requires model preparation plus
actual Vosk/MNN/MNN TTS execution evidence on a device.

The app can load the MNN runtime from `SILVERCARE_MNN_LIBRARY`, from exported
symbols already linked into the app, or from
`SilverCareiOS.app/Frameworks/SilverCareMNNRuntime.framework`,
`SilverCareiOS.app/Frameworks/libsilvercare_mnn_runtime.framework`, or
`SilverCareiOS.app/Frameworks/libsilvercare_mnn_runtime.dylib`. The simulator
and device smoke scripts also run this check against the built app bundle via
`SILVERCARE_IOS_APP_PATH`, so missing or incorrectly exported runtime symbols
cannot be hidden behind UI scaffolding. `npm run test:ios:device` performs this
bundle-runtime gate on an unsigned generic iPhoneOS build before it attempts
the signed install/launch path, so provisioning failures are separated from
runtime packaging failures.

`ios/Tools/build_mnn_runtime_wrapper.sh` builds the current device-only
`ios/Native/Vendor/SilverCareMNNRuntime.xcframework` from the vendored Alibaba
MNN 3.5.0 iOS core. `ios/Tools/build_mnn_tts_runtime_wrapper.sh` builds the
device-only `ios/Native/Vendor/SilverCareMNNTTSRuntime.xcframework` from the
vendored MNN TTS framework source. Simulator builds intentionally report no
matching MNN or MNN TTS slice until iOS simulator-compatible runtimes are
produced.

## Local MNN TTS model preparation

iOS now mirrors the Android experimental TTS model layout for
`bert-vits2-MNN`. `LocalTTSModelManager` checks the same 23 required files under:

```text
Application Support/SilverCare/multimodal_care_models/tts/bert-vits2-mnn/
```

The iOS TTS settings sheet can prepare this model bundle and reports real
`local_tts_*` readiness through the WebView bridge and local benchmark reports.
`DynamicIOSMNNTTSRuntime` can load `SilverCareMNNTTSRuntime.framework`,
`libsilvercare_mnn_tts_runtime.framework`, `libsilvercare_mnn_tts_runtime.dylib`,
or `libmnn_tts` artifacts from the app bundle, and it also checks
`SILVERCARE_MNN_TTS_LIBRARY` for simulator debugging. The current device wrapper
exports `silvercare_mnn_tts_runtime_kind`,
`silvercare_mnn_tts_voice_quality_passed`, and
`silvercare_mnn_tts_synthesize_wav`, but this remains an experimental path: a
real model synthesis run and voice-quality pass are required before it can
become a main speaking route.

## Full iOS build and device validation

Full app validation requires:

1. Full Xcode selected with `sudo xcode-select -s /Applications/Xcode.app/Contents/Developer`.
2. iPhoneOS SDK installed.
3. A real iOS device for camera, microphone, motion sensor, and TTS permission testing.
4. Local iOS model/runtime binding or DashScope credentials, depending on selected runtime mode. The iOS settings sheet can check and prepare the Qwen3-4B + DAMO-YOLO files, the Vosk Chinese ASR model, and the experimental bert-vits2-MNN TTS model under app support storage. The device MNN, MNN TTS, and Vosk wrappers are linked/embeddable, but real on-device Qwen/DAMO-YOLO inference, Vosk transcription, and MNN TTS synthesis still need validation.

Then run the device smoke from the repository root:

```sh
npm run test:ios:device
```

The smoke script first verifies the unsigned iPhoneOS app bundle contains the
MNN, MNN TTS, and Vosk iOS runtime slices, then attempts signed device
build/install/launch. If signing is not ready it writes
`ios/build/device-smoke/signing-preflight.txt` plus
`ios/build/device-smoke/summary.json`; use
`npm run check:ios:device-summary` to verify the latest device-smoke artifact.
Once signing succeeds, real device smoke testing must cover startup
permissions, camera frame processing, voice inquiry, search closure, precision
guidance, native settings changes, offline model preparation/progress
reporting, fall confirmation, TTS state callbacks, diagnostic JSONL output,
and management dashboard event generation.

For an umbrella migration gate from the repository root, run:

```sh
npm run verify:ios:migration
```

This writes `ios/build/migration-verification/summary.json` plus per-gate logs
and chains the shared checks, JavaScript tests, DashScope scenario report
validation, simulator automation, device smoke, and summary validation. While
local signing is unavailable, `SILVERCARE_IOS_ALLOW_SIGNING_BLOCKED=1` lets the
command complete with `status=blocked_by_signing` after all earlier gates pass.
Validate the generated summary with `npm run check:ios:migration-summary`.
